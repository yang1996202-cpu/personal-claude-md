# 编码纪律

这份文档用于减少 LLM 编码时最常见的错误。和项目级 `CLAUDE.md` 合并使用。

**取舍：**这些规则偏向谨慎而不是速度。非常小、可逆、低风险的任务，可以用判断力简化流程。

## 1. 先想清楚，再写代码

**不要假设。不要掩盖困惑。提前暴露取舍。**

实现前：

- 明确说出关键假设。
- 如果存在多种合理理解，不要悄悄选一个；要么说明采用哪种解释继续，要么在会影响终态时先问。
- 如果有更简单的做法，直接说出来。
- 用户方案明显会带来复杂度、风险或返工时，直接反对并给替代方案。
- 如果关键点不清楚，停下来，点名哪里不清楚，再问。

自测问题：我现在是在基于事实推进，还是在替用户脑补需求？

## 2. 简洁优先

**用能解决问题的最少代码。不要写投机代码。**

- 不加用户没要的功能。
- 不为只用一次的代码抽象一层。
- 不添加没被要求的“灵活性”“可配置性”。
- 不为现实路径到不了的场景写错误处理。
- 如果写了 200 行而 50 行能解决，停下来重写。

自测问题：一个资深工程师看到这段实现，会不会觉得它过度设计？如果会，简化。

## 3. 精准手术

**只动必须动的地方。只清理自己造成的混乱。**

编辑已有代码时：

- 不顺手“优化”相邻代码、注释或格式。
- 不重构没坏的东西。
- 匹配现有代码风格，即使你个人会用另一种写法。
- 发现无关死代码，只报告，不删除，除非用户明确要求。

如果你的改动制造了孤儿代码：

- 删除由你的改动导致的未使用 import、变量、函数或文件。
- 不删除改动前就已经存在的死代码，除非用户要求。

检验标准：每一行变更都应该能直接追溯到用户请求。

## 4. 目标驱动执行

**先定义完成标准，再循环到验证通过。**

把模糊任务转成可验证目标：

| 用户说 | 转化为 |
|--------|--------|
| “加个校验” | 先写非法输入测试，再让测试通过 |
| “修这个 bug” | 先写能复现 bug 的测试，再让测试通过 |
| “重构 X” | 确保重构前后测试都通过 |

多步骤任务使用简短计划：

```text
1. [步骤] -> 验证：[检查方式]
2. [步骤] -> 验证：[检查方式]
3. [步骤] -> 验证：[检查方式]
```

强完成标准能让 Agent 独立循环。弱完成标准，例如“让它能用”，通常需要先补清验收口径。

自测问题：我手上有没有一个能运行、能观察、能判断成败的验证方式？

## 5. 本地补充：分清应用层和基础设施层

同样是“先跑起来再调”，在不同层级后果不同：

| 层级 | 策略 | 原因 |
|------|------|------|
| 应用层：UI、普通功能、内容展示 | 可以先做小切片，再快速迭代 | 反馈快，改动成本低 |
| 基础设施层：数据流、评分、路由、配置、记忆结构、Agent 协议 | 先做简短设计，再动代码 | 地基错了会连续返工 |

涉及以下内容时，先写清楚设计判断，再实现：

- 数据流转规则：评分、路由、同步、存储格式。
- 配置体系：`settings.json`、MCP 配置、环境变量管理。
- 记忆或文档结构：标签体系、结晶规则、`MEMORY.md`。
- 多项目共享能力：skill 框架、hooks、Agent 协议。

## 6. 改动要可追溯

**动手前查重，动手时说清删了什么，动手后交代影响范围。**

- 新增功能或写新函数前，先在项目里搜一遍是否已有类似实现；找到了优先复用或扩展，不要默认从零写。
- 改动触及多个无关模块或较大范围时，先列改动计划（文件清单 + 每文件改什么 + 依赖关系）等用户确认，再动手；不硬编文件数阈值。
- 删除已有功能代码、依赖、配置时，必须明确告知用户并说理由；重构只增改不删，除非用户许可。禁止静默删除。
- 每次改动后，用一句话声明影响范围，格式如 `影响范围：XXX、YYY`。

自测问题：用户不看 diff，能不能从我口述里知道我动了什么、删了什么、影响哪里？

## 7. 调试先查链路，错误要实质处理

**修 bug 前先摸清整条链路；错误分支必须有真实处理。**

- 修 bug 前先调研完整业务链路，回答三问：完整流程是什么、影响哪些模块、别处有没有同类问题。禁止不调研就动手，禁止猜测性修复；根因无法确认时先用日志、断点或测试验证。
- 所有 try/catch 和错误分支必须有实质处理（记录日志 + 给用户可见的提示或合理降级）。禁止空 catch、禁止只 `console.log(e)`、禁止 `pass` 或 `// ignore` 静默吞错。

自测问题：我修的是症状还是根因？这个错误分支真发生时，用户和系统会怎样？

## 8. 注释是给下一次的 AI 看的

**函数注释写清背景、意图、约束，不只写功能。**

- 背景：这个函数解决什么业务问题、在什么场景被调用。
- 设计意图：为什么这样实现、放弃了哪些备选。
- 关键约束：副作用、依赖、边界条件。
- 重构时同步更新注释，不随意删已有背景/意图注释；不以“太长/自解释”为由删。

自测问题：下一次 AI（或三个月后的你）读到这个函数，能从注释理解为什么这么写吗？

这些规则有效时，应该体现为：diff 更小，没必要的改动更少，过度设计更少，关键澄清发生在实现前，而不是返工后，改动对用户全程可见。
