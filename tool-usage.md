# 工具使用纪律

## 诊断前
- 先用一句话说明当前假设，再选择最便宜的验证方式。
- 能用文件读取、`rg`、配置检查得到答案的，不要用更重的命令探索。
- 快速诊断默认在 3 个 shell 调用内给出阶段结论；复杂诊断可以继续，但要先汇报已知事实和下一步。

## 计数与核对
- 用户给出完整列表（如 `/skills` 输出、截图、命令结果）→ **直接以此为准**，不再引用之前扫描的数字。
- 核对"为什么差几个"时，逐个比对真源和文件系统，禁止心算或公式拼凑。
- 枚举完成后，用 `wc -l` 或 `grep -c` 交叉验证数量。

## 环境预检
调试依赖环境变量或外部服务的功能前，优先检查相关前置条件：
1. `env | grep RELEVANT_VAR` 或等价方式检查必需变量是否已**导出**；不要打印密钥原值。
2. `which` 检查必需二进制是否存在
3. `curl -I` 或等价命令检查服务可达性

如果前置条件明显未满足，先修环境，不要进入功能层误判。

## 范围校准
- 用户说"快速""简单""brief" → **默认上限 3 条 Bash 命令**，超出必须停下来请求许可。
- 诊断任务：3 条命令内必须给出假设，不行就换思路或汇报。
- 探索任务长时间没有外部结果时，缩小问题或给阶段结论。
- 多步骤任务开始时给简短计划；不要为了报流程拖慢执行。

## Subagent 使用规则
- 跨项目对比、大目录遍历、海量文件扫描 → 如果可用，派 sub-agent 或使用更适合的批量工具，主上下文只看摘要。
- 不要在主线程倾倒 100+ 文件的原始探索输出。

## MCP 配置规则
- MCP 配置优先检查两个位置：`~/.claude.json`（全局）或项目根目录的 `.mcp.json`（项目级）。
- 修 MCP 前必须按顺序查：全局 .claude.json → disabledMcpServers → 项目 .mcp.json。
- 不要机械跨文件复制 MCP 配置；先判断作用域和冲突。
- 发现重复条目要先报告；清理全局配置前等用户确认。

## 文档查询：优先 context7
查库、框架、SDK、API、CLI 或工具的**官方机制**（语法、配置、版本迁移、能力边界）时，优先用 context7 MCP：先 `mcp__context7__resolve-library-id` 拿 library id，再 `mcp__context7__query-docs` 查。context7 常收录官方文档镜像，比 WebSearch 更对路。

- **用 context7**：库的 API/配置/版本迁移、CLI 用法、SDK 调用、工具官方机制。
- **不用 context7**：业务逻辑调试、代码审查、重构、通用编程概念、项目特有约定——这些读代码和项目文档。
- 注意：context7 的官方仓库源里可能夹带示例代码（样例，非实现），别当成核心机制。
